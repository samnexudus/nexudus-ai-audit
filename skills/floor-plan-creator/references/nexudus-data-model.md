# Nexudus Floor Plan Layout Data Model

Reference for the geometric shape Nexudus expects. The cleaned JSON produced by `normalise.py` matches this shape.

## Units

- **All coordinates and lengths are in millimetres.** A 9.8m wall has `length: 9800`.
- **Angles are degrees (0–360).** `0` = pointing along +x axis (east). `90` = pointing along +y axis (south, since y grows downward in image coordinates). `180` = west. `270` = north.
- **Area sizes are in square metres** (not mm²).

## Top-level cleaned JSON

```json
{
  "name": "First Floor",
  "floorLevel": 0,
  "nodes": [...],
  "edges": [...],
  "areas": [...]
}
```

## Nodes

A node is a wall corner / intersection point.

```json
{
  "key": "550bf783-e76d-4192-b9b8-62ed25517cdb",
  "x": 33300,
  "y": 40300
}
```

- `key` — UUIDv4 string. Generated by `normalise.py`. Used by edges (`source`/`target`) and areas (`nodes`).
- `x`, `y` — integer millimetres. Origin is arbitrary (typically the top-left of the source image).

## Edges

An edge is a wall segment between two nodes.

```json
{
  "key": "b7dba1f0-4384-4fe9-8283-05aca95e2474",
  "source": "550bf783-...",
  "target": "4dc16a27-...",
  "angle": 270,
  "length": 9800,
  "width": 200,
  "height": 2600,
  "isPartition": false,
  "openings": []
}
```

- `key` — UUIDv4.
- `source`, `target` — node UUIDs. The edge runs from source to target. Direction matters because `angle` describes the source→target direction.
- `angle` — degrees, see Units above.
- `length` — millimetres. Should equal Euclidean distance between source and target nodes.
- `width` — wall thickness in mm. Default `200` (a typical interior wall).
- `height` — wall height in mm. Default `2600` (~2.6m, standard ceiling).
- `isPartition` — `false` for a structural wall, `true` for a half-height divider.
- `openings` — empty in v1. Doors and windows would go here as separate `FloorPlanLayoutOpening` entities attached to this edge.

## Areas

An area is an enclosed room defined by an ordered list of node UUIDs.

```json
{
  "name": "Staff Workstations",
  "size": 79.68,
  "color": "#ffffff",
  "nodes": [
    "c97855aa-...",
    "c82f4b9b-...",
    "cbcaffed-...",
    "532c23ca-...",
    "4fb92cf8-...",
    "9f2392ca-...",
    "cad34035-..."
  ],
  "containedAreas": []
}
```

- `name` — display label.
- `size` — area in **square metres** (not mm²). Computed by the shoelace formula on the node polygon, divided by 1,000,000.
- `color` — hex string. Default `#ffffff`.
- `nodes` — ordered polygon vertices (CW or CCW, consistent within the file). Must reference existing node UUIDs.
- `containedAreas` — empty in v1 (used for nested zones like "Hot Desks within Open Office").

## Defaults applied by `normalise.py`

| Field | Default |
|---|---|
| `edge.width` | 200 |
| `edge.height` | 2600 |
| `edge.isPartition` | `false` |
| `area.color` | `#ffffff` |
| `floorLevel` | `0` |

These match the example payload provided as ground truth.

## Coordinate origin convention

Image coordinates use top-left origin, y growing downward. The cleaned JSON preserves this convention because Nexudus's example payload appears to do the same (node `y` values increase moving towards the bottom of the floor plan in the source PNG).

## Wall thickness and shared walls

Two adjacent rooms share a wall — they should both reference the **same two nodes** for that wall, not two parallel sets. The 200mm wall thickness is a render-time concept; the underlying graph treats walls as zero-thickness lines between nodes.
